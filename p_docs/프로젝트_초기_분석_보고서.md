# SELab Block Editor — 프로젝트 초기 분석 보고서

> 근거: 저장소 `README.md`, `package.json`, `media/editor`·`src/panel` 핵심 모듈 스캔 (2026-05-14 기준)

---

## 1. 요약

**SELab Block Editor**는 VS Code 확장으로, SysML v2 BDD 스타일의 블록 구조를 **JSON → 웹뷰(mxGraph + ELK)** 파이프라인으로 보여주는 뷰어입니다. SELab 미니 프로젝트의 핵심 과제는 **`tests/*.json` 렌더 시 직교(orthogonal) 엣지 라우팅·노드 배치 품질**을 README에 적힌 기준에 맞게 끌어올리는 것입니다.

현재 README의 As-Is 설명과 일치하도록, 개선 포인트는 **레이아웃(ELK 그래프 구성·옵션·계층 반영)**과 **엣지 경로(직교·교차·노드 통과)**가 한 축으로 묶여 있습니다.

## 2. 프로젝트 목적과 평가 기준

| 구분 | 내용 |
| --- | --- |
| 제품 역할 | JSON 기반 블록 다이어그램 뷰어 (VS Code WebView) |
| 과제 핵심 | 엣지 직교 정렬 품질 극대화 + 계층·간격·대칭 등 레이아웃 품질 |

README에 명시된 품질 기준(교차 최소화, 노드 겹침 없음, 엣지-노드 비중첩, bend 최소, 균일 간격, 계층 배치, 종단 명확성, 형제 대칭)은 이후 작업의 **체크리스트**로 두는 것이 좋습니다.

## 3. 기술 스택과 디렉터리 역할

| 영역 | 기술 | 역할 |
| --- | --- | --- |
| 확장 호스트 | Node + VS Code API | 커맨드 등록, WebView 패널, JSON 로드 |
| 웹뷰 렌더 | mxGraph (`USE_MXGRAPH = true` in `boot.js`) | 셀·엣지 실제 그리기 |
| 자동 배치 | elkjs (`media/editor/layout/elkLayout.js`) | 노드 좌표·엣지 라우팅(직교 옵션 등) |
| 설정 집약 | `media/editor/config/displaySettings.js` | ELK·노드 크기·간격 등 상수의 단일 진실 공급원 |
| 블록 필터 | `src/panel/BlockModelBuilder.js` | 블록 다이어그램에 맞는 노드/엣지 `kind` 화이트리스트로 모델 축소 |
| 빌드 | esbuild (`scripts/build.mjs`) | `src/extension.js` → `dist/extension.js` |

README의 구조 설명은 대체로 유효하나, 실제 저장소에는 `tests/test-3.json` ~ `test-10.json` 등 추가 픽스처가 있어 **회귀 검증용으로 활용**할 수 있습니다.

## 4. 런타임 데이터 흐름 (개념)

1. 사용자가 JSON을 열고 **블록 다이어그램 옆에 열기** / 명령 팔레트로 패널 오픈.
2. 확장이 JSON을 웹뷰로 전달 → 패널 쪽에서 모델 정규화·블록 필터링 (`BlockModelBuilder` 등).
3. 웹뷰 `boot.js`에서 정규화 모델을 유지하고, **`applyElkLayout`**으로 좌표·연결 라우팅을 갱신.
4. **mxGraph** 팩토리로 모델을 다시 그림 (`renderModel` 등).

과제 성패는 **3번(ELK 입력 그래프 설계 + 옵션)과 4번(렌더가 ELK 결과를 왜곡 없이 반영하는지)**에 많이 좌우됩니다.

## 5. As-Is 문제에 대한 코드 관점 가설

README에서 지적한 현상(세로로 길게 쌓임, 캔버스 밖 노드, 엣지가 무관 노드 위 통과, 상단에 떠 있는 노드, 계층이 안 보임)은 단일 버그라기보다 아래가 **복합**된 결과일 가능성이 큽니다.

1. **ELK에 넘기는 그래프의 parent/children 구조**가 SysML containment·specialization 의미와 완전히 맞지 않거나, 일부 엣지가 **계층이 아닌 일반 엣지**로만 처리되어 layered 알고리즘이 기대한 트리/컴파운드를 만들지 못함.
2. `elkLayout.js`의 `isHierarchicalEdgeKind` 등 **엣지 종류별 분기**가 실제 JSON의 `kind` 문자열과 미세하게 어긋나면, containment 전용 처리가 빠질 수 있음.
3. **노드 크기 사전 계산**(`displaySettings`의 `nodePrecompute` 등)과 ELK spacing이 불일치하면 겹침·밀림·캔버스 이탈이 발생할 수 있음.
4. 엣지는 `drawEdge.js` 등에서 **소스/타겟 해석(이름·접미사·액티비티 특수 케이스)**이 복잡해, 블록 뷰에서는 간혹 의도와 다른 앵커가 잡힐 여지가 있음(블록 과제 범위에서는 우선 ELK 라우팅 결과를 신뢰하는지 확인).

검증 시에는 **브라우저/웹뷰 콘솔**에서 ELK 실패 시 `fallbackGrid`로 떨어지는지(`elkLayout.js` 상단 로그)도 확인하는 것이 좋습니다.

## 6. 과제 풀이 방향 — 권장 로드맵

아래 순서는 리스크를 낮추면서 README 기준에 맞추기 위한 **실무형 단계**입니다.

### 6.1 기준선 고정

- `tests/test-1.json`을 **골든 케이스**로 고정하고, 변경 전후 스크린샷 또는 `npm` 스크립트(`test:block` 계열)로 회귀 범위를 정함.
- `displaySettings.js`의 `elk` 블록을 표로 정리해, 한 번에 바꾸는 옵션 수를 제한(디버깅 가능성 확보).

### 6.2 ELK 입력 그래프 정합성 우선

- containment / specialization / featuretyping / association이 **ELK의 children vs edges**로 어떻게 매핑되는지 `elkLayout.js` 전체 흐름을 추적.
- README 목표인 **계층적 배치**를 위해, containment를 **실제 parent-child**로 올리는 경로가 있는지, 없다면 추가하는 편이 효과가 큰 경우가 많음.

### 6.3 알고리즘·옵션 튜닝

- 기본은 `layered` + `DOWN` + `ORTHOGONAL` 라우팅이 이미 잡혀 있음 — **교차 최소화**(`crossingMinimization`), **층 간 간격**(`nodeNodeBetweenLayers`, `edgeNodeBetweenLayers`), **컴팩션**(`compactionStrategy`, `connectedComponents`)을 단계적으로 조정.
- 그래프가 크고 association이 많으면 **한 알고리즘으로 모든 품질 기준을 동시 만족**하기 어려울 수 있음 → 필요 시 **엣지 레이어링**(예: hierarchy 전용 패스 vs association 패스) 같은 설계 검토는 후순위로 두고, 먼저 단일 ELK 그래프에서 한계를 측정.

### 6.4 엣지-노드 비중첩·교차

- ELK의 `elk.spacing.edgeNode` / `edgeEdge` 및 라우팅 제약이 충분한지 확인.
- 여전히 무관 노드 관통이면 **obstacle(장애물) 제약** 또는 **포트/사이드 고정** 같은 ELK 고급 옵션, 또는 **라우팅 후처리**(프로젝트 정책상 허용된다면)를 검토.

### 6.5 mxGraph 측 일치

- ELK가 준 `bendpoints`/`sections`가 mxGraph 엣지 기하에 그대로 반영되는지 `MxCellFactory`·엣지 생성 코드를 대조.
- 좌표 스케일·패딩·컨테이너 헤더 높이(`displaySettings`의 container/compartment) 불일치가 **시각적 “노드 통과”**로 보이는 경우가 있음.

### 6.6 제출 전 품질 게이트

- README 체크리스트를 짧은 **자체 평가표**로 두고, `test-1` 외 `test-3` 등 1~2개 추가 파일로 과적합 방지.

## 7. 리스크와 주의사항

- **ELK 미로딩 시** `fallbackGrid`만으로는 과제 품질 목표 달성이 어려움 — 개발 호스트에서 ELK 번들 로드 여부를 반드시 확인.
- `BlockModelBuilder`가 필터링하는 `kind` 집합 밖 노드/엣지는 화면에 안 나올 수 있음 — 디버깅 시 “JSON에는 있는데 안 보인다” 착시 방지.
- 사용자 규칙(코딩 스타일)과 별개로, **본 저장소의 `media/editor`는 IIFE·전역 `window.SELAB` 패턴**이므로 새 코드도 기존 패턴에 맞추는 것이 안전함.

## 9. 결론

이 프로젝트는 **“뷰어 한 기능”이 아니라 레이아웃·라우팅 품질을 끌어올리는 알고리즘 과제**에 가깝습니다. README의 목표를 달성하려면 **ELK 입력(계층 구조) 정합성 → 옵션 튜닝 → 렌더러와의 기하 일치** 순으로 접근하고, 중간마다 `test-1` 기준선으로 시각적 회귀를 잡는 것이 가장 비용 대비 효과가 큽니다.

이후 `p_docs`에 **옵션 실험 로그**, **ELK 그래프 덤프(JSON)** 같은 산출물을 덧붙이면 제출·면접 설명용으로도 재사용하기 좋습니다.
