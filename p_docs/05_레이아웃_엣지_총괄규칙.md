# 레이아웃·앵커·엣지 꺾임 — 총괄 규칙 (코드 전수 기준)

본 문서는 **BDD 블록 다이어그램을 mxGraph로 그리는 경로**만 대상으로, 저장소에 실제로 있는 코드를 읽고 정리했습니다.  
**최종 기준은 루트 `README.md`**입니다.

## 1. 데이터가 흐르는 순서 (한 줄 타임라인)

1. JSON → `BlockModelBuilder.js` — 노드/엣지 필터, `nestedSpecChrome` 플래그, 다중 containment 시 `id__in__parent` 복제(모델).
2. WebView `normalizeModel` — `nodeTransformer` / `edgeTransformer`, `nestedSpecChrome.applyNestedSpecChrome`.
3. `precomputeNodeSizes` (`layout.js`) — 헤더 폭(이름·🔗·접기) 반영 후 `applyElkLayout` (`elkLayout.js`).
4. `bdd.applyPostLayout` (`bddLayout.js`) — spec 밴드·형제 대칭·겹침·마진, **spec waypoint 삭제**.
5. `MxCellFactory.renderModel` — `createVertex` → 리사이즈 → `createEdge`(nested spec 선 **생략**) → `refresh` → `rerouteAllEdges` / `distributeOverlappingEdges`.

**노드 좌표의 “진실”**은 2~3단계의 **인메모리 `elements[]`**이고, **선 경로의 “진실”**은 4단계에서 **mx `geometry` + 스타일 앵커**입니다.

## 2. 노드 배치 규칙 (어디서 무엇이 정해지나)

### 2.1 ELK (`elkLayout.js`)

- **입력:** `diagramData.elements` (id, parent, x, y, width, height …), `connections`.
- **알고리즘:** 기본 `layered`, 방향 `DOWN`, `elk.edgeRouting` 등은 **`displaySettings.elk`**와 병합.
- **컨테이너:** `elements`의 **parent 체인**으로 그룹을 만들고, ELK 하위 그래프로 배치.
- **ELK 그래프에 넣는 엣지:**  
  - **제외:** `containment` 등 계층형(`isHierarchicalEdgeKind` + `kindClass` 없음), **`specialization` 전부**, 교차 컨테이너 `featuretyping`(일부 예외), 기타 필터.  
  - **포함:** association·dependency 등 비계층 연결, 같은 부모 `flow`, `body`↔succession 가상 엣지 등 (코드 블록 `allElkEdges`).
- **spec 전용 후처리 (같은 파일 내):** `applySpecializationVerticalLayout` — spec에 연루된 **루트 노드**의 **Y 밴드**를 다시 맞춤 (부모가 자식보다 위로 오도록).

### 2.2 BDD 후처리 (`bddLayout.js` — `applyPostLayout` 호출 순서)

`bddLayout.js`의 `applyPostLayout`은 **항상 아래 순서**입니다.

1. **`applySpecVerticalBands`** — spec을 **무방향 연결 요소**로 나눈 뒤 요소마다 **레이어별 Y 밴드** 한 번만 정렬(다중 상속 시 동일 노드가 루트별로 두 번 밴딩되지 않게 함, `displaySettings.bdd.specLayerGap` 등).
2. **`applySpecChildrenSymmetric`** — **루트에만 있는** spec 부모마다, 그 **직접 spec 자식**을 부모 **아래 한 줄**에 이름순 정렬, 부모 중심 기준 **좌우 대칭** (`specSiblingGap`, `specCorridorGap` 반영한 `belowParentGap`).
3. **`fitDiagramToMargins`** — 전체 bbox를 `diagramMargin` 안으로 밀기.
4. **`resolveSiblingOverlaps`** — **같은 parent**를 가진 형제들 bbox 겹침을 **최대 16패스**까지 x/y로 밀어 해소 (`siblingOverlapGap`).
5. **`clearSpecWaypoints`** — spec 연결의 `waypoints` 삭제 → **화면 선은 spec 라우터가 전담**.

### 2.3 mxGraph에 노드 좌표가 들어갈 때 (`MxVertexBuilder.js`)

- **루트 노드:** `node.x`, `node.y`(또는 `relativeX`/`relativeY`)를 **그래프 defaultParent 기준** 로컬 좌표로 `insertVertex`.
- **자식 노드:** 부모 셀 아래에 붙이고, **`relativeX` / `relativeY`**(ELK가 채운 값)를 사용. `geometry.relative = false`로 **픽셀 오프셋**으로 둠.

### 2.4 렌더 직후 (`MxCellFactory.js` — `renderModel`)

- 모든 노드 `ensureCell` 후 **`resizeParentsToFitChildren`** — 컨테이너가 자식+푸터를 덮도록 크기 조정.
- **`attachFeatureTypingFooters`** 등 부가 셀.

## 3. 엣지 종류별 “누가 그리나”

| 종류 | 생성 | 경로·앵커 |
|------|------|-----------|
| **specialization** (일반) | `createEdge` → `specEdgeRouter.routeSpecEdge` | **`specEdgeRouter.js`** (N→S, ELK waypoint 없음) |
| **specialization** (nested spec 크롬) | `nestedSpecChrome` — **엣지 셀 없음** | 헤더 **🔼** (`MxLabelUtils`), 선택 시 부모 HL (`MxNeighborHighlight`) |
| **그 외** (association 등, 모델에 남은 경우) | `MxEdgeBuilder.routeEdge` | ELK `waypoints`가 있고 `isPathReasonable`이면 그 점 사용, 아니면 **`buildOrthogonalPath`** |
| **containment** | `MxEdgeBuilder`에서 **엣지 셀 생성 안 함** (`createEdge` 초반 return) | 포함 관계는 **노드 parent**로만 표현 |

## 4. 앵커(테두리 붙는 방향) 규칙

### 4.1 비-spec (`MxEdgeBuilder.js` — `resolveSides`)

- **specialization이면** (실제로는 spec은 여기 안 탐): 주석대로 **N 출구 / S 진입** 고정.
- **일반:** 소스·타깃 **중심점**으로 벡터 `dx`, `dy` 계산.
  - `|dy| ≥ |dx|` 이면 **수직:** 타깃이 아래면 `S→N`, 위면 `N→S`.
  - 아니면 **수평:** 타깃이 오른쪽이면 `E→W`, 왼쪽이면 `W→E`.

### 4.2 spec (`specEdgeRouter.js` — `resolveSpecSides`)

- **일반 spec 엣지:** **N 출구 → S 진입** 고정. 동적 4방향·형제 S 출구는 **롤백**.
- **nested spec 크롬** (`child.parent === spec.target`): **엣지 없음**, 헤더 **🔼**, 선택 시 부모 HL.

`routeSpecEdge`는 **`buildSpecPath` + N/S**만 사용합니다. 소스에 `buildNestedEscapePath` 등이 남아 있어도 **마무리 시점 분기에는 연결되지 않음** (과거 E/W 탈출 실험 잔재).

앵커는 mx 스타일 **`exitX`/`exitY`/`entryX`/`entryY`** + **`setConnectionConstraint`**(정규화된 면+비율)로 고정합니다.

### 4.3 같은 면에 선 여러 개 (`specEdgeRouter.distributeSpecEdges`)

1. **루트 아래 spec** (`isRootChildBelowParent` + N/S): `alignRootBelowParentAnchors` — **진입 S**를 **자식 중심 X**에 맞춤, **출구 N**은 0.5.
2. 나머지: 같은 타깃·같은 **entry 쪽**끼리, 같은 소스·같은 **exit 쪽**끼리 **정렬 후** `frac = (i+1)/(n+1)`로 **테두리 위치 분산**.
3. 마지막에 **`rerouteAllSpecEdges`**로 경로 재계산.

## 5. 엣지가 “어디서 꺾이는지” (기하 규칙)

### 5.1 좌표계 (공통)

- 경로 점은 **절대 좌표(다이어그램 기준)**로 계산한 뒤, mx `geometry.points`에는 **엣지 셀의 부모 체인 원점**(`edgeControlOrigin` / `specEdgeRouter` 동일 개념)을 빼서 **상대 좌표**로 넣습니다. **소스 노드 원점 기준이 아닙니다.**

### 5.2 비-spec (`MxEdgeBuilder.buildOrthogonalPath`)

- **출발점** `exitPt` = 소스 bbox의 **exitSide** 위 `exitFrac` 지점.
- **도착점** `entryPt` = 타깃 bbox의 **entrySide** 위 `entryFrac` 지점.
- **둘 다 수직면(N/S):**  
  - x 같으면 **직선 1구간**.  
  - 다르면 **수평 채널 `yCh`**: `verticalChannelY` — 자식 위·부모 아래 **갭 중앙**, 겹치면 **부모 밖 + APPROACH_PAD(16px)**.
- **둘 다 수평면(E/W):** `horizontalChannelX`로 **수직 채널 `xCh`**.
- **한쪽만 수직:** 나머지 한 번 수평/수직 보정 (`exitVert` 분기).

스타일: **`orthogonalEdgeStyle`**, 중간점 있으면 `geometry.points`에 상대 좌표. **중간점이 없으면** mx가 터미널만 이어 **사실상 직선에 가까울 수 있음** (대각선처럼 보이는 경우).

### 5.3 spec (`specEdgeRouter` — `routeSpecEdge`)

- **`buildSpecPath`** + **N/S** 앵커, 장애물 시 `refineOrthogonalPath`·`sanitizeNsSpecPathNoFaceRide` 등.
- **nested spec 크롬**은 이 경로를 타지 않음 (엣지 셀 없음).

적용: **`segmentEdgeStyle` + `noEdgeStyle=1`**, `geometry.points` = 중간점만(끝점 제외). **2점만이면** `points` null → 터미널만 이어질 수 있음.

## 6. 렌더 이후 파이프라인 (`MxCellFactory.js`)

```
renderModel 끝 → graph.refresh()
→ rerouteAllEdges(): 그래프를 **한 번** DFS하며 엣지마다 spec이면 `routeSpecEdge`, 아니면 `routeEdge`
→ distributeOverlappingEdges(): spec은 distributeSpecEdges(내부에서 `rerouteAllSpecEdges`로 spec만 재라우팅), 비-spec은 앵커 분산 후 routeEdge
→ graph.refresh()
```

## 7. 설정값이 규칙에 미치는 영향

| 설정 위치 | 영향 |
|------------|------|
| `displaySettings.elk.*` | ELK 노드 간격, 레이어 간격, `edgeRouting`, 교차 최소화 등 |
| `displaySettings.bdd.*` | spec **수직 밴드 간격**, 형제 간격, 통로·마진, 형제 겹침 갭 |
| `displaySettings.nodeLabel.*` | 접기·연관 링크 **헤더 폭** (`foldButtonReservePx` 등) — `layout.js` precompute |
| `MxEdgeBuilder` 상수 `APPROACH_PAD` (16) | 직교 채널이 부모 밖으로 나갈 때 여유 |
| `MxStyleManager` 기본 엣지 | `orthogonalEdgeStyle`, `rounded=1` 등 **비-spec 초기 스타일** |

## 8. 한 페이지 요약 표

| 질문 | 답 (현 코드) |
|------|----------------|
| 노드가 왜 여기 있나? | ① ELK 전역 ② `applySpecializationVerticalLayout` ③ `bddLayout` spec 밴드·형제·겹침·마진 |
| 상속 선은 누가 그리나? | **`specEdgeRouter.js`만** (ELK 엣지·waypoint 없음) |
| 상속 앵커 방향은? | **N→S** (일반 spec) / **포함=상속 부모**는 **선 없음·🔼** |
| 다중 부모 자식 UI는? | **미해결(보류)** — `07_마무리_및_알려진_한계.md` §2 |
| 비-spec 앵커는? | **중심 벡터**로 N/S 또는 E/W |
| 꺾임은 어디서 정해지나? | spec: **명시 점** + segment / 비-spec: **buildOrthogonalPath** 또는 ELK waypoint |
| 왜 ELK ORTHOGONAL만으로 안 끝나나? | spec·containment 등 **대부분의 BDD 선은 ELK 엣지 라우팅 대상이 아님** |
